# 项目介绍与核心问题

## 1. 项目概述

### Q: ForgeAgent 是什么？

**ForgeAgent** 是面向开发场景的 AI Agent 应用，采用 **Plan-and-Execute** 认知框架，具备规划、记忆、工具、执行四大核心能力。

```
用户输入 → Planner(规划) → Actor(执行) → Learner(反思)
                              ↑              │
                              └──── replan ──┘
```

### Q: 技术栈？

| 层级 | 技术选型 | 说明 |
|------|----------|------|
| 后端框架 | **FastAPI** | 异步高性能 API 框架 |
| Agent 编排 | **LangGraph** | 基于 LangChain 的状态机图编排 |
| LLM 调用 | **LangChain Core** + OpenAI | 统一的 LLM 接口抽象 |
| 数据库 | **SQLite** (业务) + **SQLite** (checkpoint) | 轻量级持久化 |
| 前端 | **React 18** + TypeScript | SSE 流式交互 |
| 向量存储 | **ChromaDB** / FAISS | RAG 混合检索 |

### Q: 为什么不是通用框架？

区别于 LangChain/LangGraph 等通用框架，ForgeAgent 是针对特定场景优化的应用：
- 内置 Plan-Act-Learn 工作流
- 开箱即用的记忆系统
- 内置工具注册表
- 前端 SSE 监控界面

## 3. 核心问题 10 题

### Q1: Plan-and-Execute vs ReAct 区别？

| 模式 | 特点 | 适用场景 |
|------|------|----------|
| **ReAct** | 思考-动作-观察循环，每步立即执行 | 简单任务、工具调用 |
| **Plan-and-Execute** | 先规划再执行，计划驱动 | 复杂任务、多步骤 |
| **ForgeAgent** | Plan-Act-Learn，加入反思机制 | 开发任务、重规划 |

**ForgeAgent 的三角循环**：
```
Plan → Act → Learn
 ↑              │
 └── replan ────┘
```

### Q2: 什么是黑板记忆？

跨任务共享的反思笔记池：
- 每个任务结束后 Learner 生成反思
- 追加到 `blackboard_notes`
- 下个任务的 Planner 读取最后 10 条
- 实现"经验积累"

```python
# 黑板数据结构
blackboard_notes: list[str]  # 反思要点列表

# 写入选入
notes.append(reflection_text)
notes = cap_blackboard_notes(notes, max_notes=64)

# 规划时读取
notes = state.get("blackboard_notes") or []
tail = notes[-10:]  # 取最后 10 条
```

### Q3: 为什么需要重规划？

当执行失败或 LLM 反思认为需要调整计划时：
```python
# 重规划触发条件
should_replan = (
    not failed                              # 非失败
    and can_replan                          # 还有重规划次数
    and (actor_replan or llm_request_replan)  # 有人请求
)
```

### Q4: SSE vs WebSocket 选型？

| 特性 | SSE | WebSocket |
|------|-----|-----------|
| 方向 | 单向服务端推送 | 双向 |
| 实现 | 简单 | 复杂 |
| 重连 | 自动 | 需手动 |
| 适用 | 事件流推送 | 实时交互 |

ForgeAgent 选 SSE：事件流推送场景简单够用。

### Q5: 为什么用 SQLite？

MVP 阶段：
- 零配置、零运维
- 异步 SQLAlchemy 2.0 够用
- 后续可平滑迁移 PostgreSQL

### Q6: Session vs Task 区别？

| 概念 | 说明 |
|------|------|
| **Session** | 会话，一次对话上下文 |
| **Task** | 任务，Session 内的具体任务 |
| **Message** | 消息，Session 内的对话记录 |
| **TaskEvent** | 事件，Task 执行过程中的事件流 |

```
Session ───┬── Task ─── Event
           └── Message
```

### Q8: 工具注册表设计？

```python
class ToolRegistry:
    def __init__(self):
        builtins = list_builtin_tools()
        self._tools = builtins

    async def refresh(self, settings):
        # 合并内置 + MCP 工具
        mcp_part = await tools_from_mcp_settings(settings.mcp)
        self._tools = self._merge((builtins, mcp_part))

    async def execute(self, name, args):
        item = self._by_name.get(name)
        if item.source == "builtin":
            return await execute_builtin(name, payload)
        if item.source == "mcp":
            return await self._execute_mcp(item, payload)
```

### Q9: 为什么需要熔断器？

防止连续失败时持续调用：
```python
class CircuitBreaker:
    CLOSED = "closed"      # 正常，允许请求
    OPEN = "open"          # 熔断，拒绝请求
    HALF_OPEN = "half_open" # 半开，允许探测请求

    # 失败 10 次后熔断
    # 60 秒后进入半开状态
    # 半开成功恢复，失败继续熔断
```

### Q10: Checkpoint 作用？

LangGraph checkpoint 实现：
- 任务中断恢复
- 状态持久化
- 并发控制

```python
# 独立 SQLite 存储
saver = AsyncSqliteSaver(conn)
graph = builder.compile(checkpointer=saver)

# 恢复执行
snap = await graph.aget_state(config)
if snap.next:  # 有未完成节点
    result = await graph.ainvoke(None, config)
```

## 5. 关键技术点速记

### Checkpoint 持久化

- 独立 SQLite 文件，与业务库分离
- task_id 作为 thread_id
- 服务重启后恢复率达 99%+

### RAG 混合检索

- 向量检索 (ChromaDB) + BM25 关键字检索
- RRF (倒数排名融合) 算法合并结果
- Cross-Encoder Rerank 重排序

### Token 管理

- tiktoken 精确计数
- 自新向旧贪心截断
- 对话摘要压缩超长会话

### 熔断器

- 三态机：CLOSED → OPEN → HALF_OPEN → CLOSED
- 失败累加达到阈值触发
- 超时后进入半开探测
