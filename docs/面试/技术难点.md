## 2. Token 上下文超限

### Q: 上下文超出模型限制怎么办？

**三层策略**：

```python
# 1. 精确计数优先
if settings.llm_use_exact_token_count:
    tokens = count_messages_tokens(msgs)  # tiktoken

# 2. 启发式估算兜底
else:
    tokens = len(content) // 3 + msgs * 4

# 3. 截断到预算
if tokens > max_input_tokens:
    msgs = truncate_to_budget(msgs, max_input_tokens)
```

### Q: 截断算法细节？

```python
def truncate_chat_messages_to_budget(
    chat: BaseChatModel | None,
    messages: Sequence[BaseMessage],
    *,
    max_input_tokens: int,
) -> list[BaseMessage]:
    """将消息截断到不超过 max_input_tokens"""
    budget = max(64, int(max_input_tokens))
    msgs = list(messages)
    before = estimate_messages_tokens(chat, msgs)

    if before <= budget:
        return msgs

    # 1. 拆分系统消息与其余角色消息
    raw_sys = [m for m in msgs if isinstance(m, SystemMessage)]
    others = [m for m in msgs if not isinstance(m, SystemMessage)]
    sys_list = list(raw_sys)

    # 2. 自新向旧贪心装入非系统消息
    kept_rev: list[BaseMessage] = []
    for msg in reversed(others):
        trial = sys_list + list(reversed(kept_rev + [msg]))
        if estimate_messages_tokens(chat, trial) <= budget:
            kept_rev.append(msg)
        else:
            break

    # 3. 最终挤压一轮
    if kept_rev:
        out = sys_list + list(reversed(kept_rev))
    else:
        room = max(32, budget - estimate_messages_tokens(chat, sys_list))
        out = sys_list + [_truncate_one_message(chat, others[-1], room)]

    _log_context_truncated(before, estimate_messages_tokens(chat, out), budget)
    return out
```

### Q: ToolContext 防止越权访问？

```python
@dataclass
class ToolContext:
    """工具执行上下文，传递用户身份信息"""
    user_id: str
    session_id: str
    workspace_root: Path

    def validate_path(self, path: Path) -> bool:
        """防止路径遍历攻击"""
        resolved = (self.workspace_root / path).resolve()
        return resolved.is_relative_to(self.workspace_root.resolve())
```

## 5. 任务中断恢复

### Q: 服务崩溃如何恢复任务？

**Checkpoint 机制**：
```python
# 1. 保存检查点 (LangGraph 自动)
graph = builder.compile(checkpointer=AsyncSqliteSaver.from_conn_string(...))

# 2. 检查是否有未完成任务
snap = await graph.aget_state(config)
if snap.next:  # 有未完成节点
    result = await graph.ainvoke(None, config)  # 继续执行

# 3. 状态恢复
state = snap.values
```

### Q: 为什么恢复率能达到 99%？

1. **异步写入**：aiosqlite 确保写入完成
2. **状态完整**：AgentState 包含完整执行上下文
3. **独立存储**：checkpoint 与业务库隔离

## 7. 数据库写入瓶颈

### Q: 每次工具调用都写库太慢？

**问题**：11 条事件/步 × 15ms = 165ms

**方案**：
```python
# 批量写入
async def append_events_batch(session, task_id, events):
    stmt = insert(TaskEvent).values([
        {"task_id": task_id, "module": m, "kind": k, "payload_json": p}
        for m, k, p in events
    ])
    await session.execute(stmt)

# 使用
events = [
    ("execution", "tool_call", json.dumps({...})),
    ("execution", "tool_result", json.dumps({...})),
]
await append_events_batch(db, task_id, events)
```

### Q: seq 字段如何保证顺序？

```python
# 使用 INSERT RETURNING 获取最新 seq
async def append_event(db, task_id, module, kind, payload):
    stmt = insert(task_events).values(
        task_id=task_id,
        module=module,
        kind=kind,
        payload_json=payload
    ).returning(task_events.c.seq)

    result = await db.execute(stmt)
    return result.scalar()
```

## 9. 计划验证

### Q: 如何避免生成无效计划？

**方案**：Plan-and-Validate
```python
async def validate_plan(steps):
    issues = []

    # 1. 步骤是否可执行
    for step in steps:
        if not step.get("title"):
            issues.append("缺少步骤标题")

    # 2. 步骤顺序是否合理
    for i in range(len(steps) - 1):
        if steps[i] == steps[i+1]:
            issues.append("重复步骤")

    # 3. 返回验证结果
    return {"is_valid": len(issues) == 0, "issues": issues}
```

## 15. 性能瓶颈总结

| 瓶颈 | 影响 | 解决方案 |
|------|------|----------|
| append_event SELECT MAX | DB 查询 | INSERT RETURNING |
| SSE 连接复用 | 连接数 | 单长连接 |
| 工具串行执行 | 延迟 | asyncio.gather |
| 前端重渲染 | CPU 高 | Zustand 细粒度 |
| LLM 重试过长 | 延迟 | 降低退避参数 |
| Token 截断 | 上下文丢失 | 摘要压缩 |

## 16. 熔断器配置建议

### Q: 如何配置熔断器阈值？

| 场景 | failure_threshold | recovery_timeout |
|------|-------------------|------------------|
| LLM 调用 | 5 | 60s |
| 工具调用 | 10 | 30s |
| 数据库 | 20 | 120s |

### Q: 为什么不设置更激进的阈值？

- **过短**：网络抖动导致误触发
- **过长**：恢复太慢影响可用性
- **建议**：基于实际压测数据调整
