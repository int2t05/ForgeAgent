### Q: 为什么选择分层架构？

| 分层 | 职责 | 好处 |
|------|------|------|
| API | 请求路由、参数校验 | 解耦、可复用 |
| Service | 业务编排、事务边界 | 事务一致 |
| Agent | LLM 调用、状态机 | 核心逻辑独立 |
| Repository | 数据访问 | 数据库解耦 |

### Q: 为什么用异步？

```python
# 同步：阻塞等待
result = await chat.ainvoke(messages)  # 等待 LLM 时释放线程

# 异步：释放线程，可并发处理多个请求
async def handle_request():
    task1 = asyncio.create_task(run_agent(task_id_1))
    task2 = asyncio.create_task(run_agent(task_id_2))
    await asyncio.gather(task1, task2)
```

场景：
- LLM 调用慢（1-3 秒）
- DB 操作可并发
- SSE 轮询长连接

## 2. LangGraph Checkpoint 持久化

### Q: Checkpoint 是如何实现的？

基于 LangGraph 的 `AsyncSqliteSaver`，关键设计：

```python
# backend/app/modules/workflow/checkpointer.py

async def _open_sqlite_checkpointer(db_path: str) -> AsyncSqliteSaver:
    """SQLite：确保目录存在，建立长连接并初始化表结构"""
    path = Path(db_path)
    path.parent.mkdir(parents=True, exist_ok=True)
    conn = await aiosqlite.connect(str(path.resolve()))  # 建立 aiosqlite 长连接
    saver = AsyncSqliteSaver(conn)
    await saver.setup()  # 初始化
    return saver
```

### Q: 为什么选择独立 SQLite？

1. **隔离**：checkpoint 是 LangGraph 内部结构，与业务表分离
2. **性能**：避免与业务 SQLAlchemy 混合使用导致的连接竞争
3. **迁移**：未来可切换到 PostgreSQL（只需改配置）

```python
async def open_langgraph_checkpointer(settings: Settings) -> BaseCheckpointSaver:
    if settings.langgraph_checkpoint_postgres_uri:
        return await _open_postgres_checkpointer(...)  # 生产环境可选
    return await _open_sqlite_checkpointer(...)  # 默认 SQLite
```

### Q: 服务重启后如何恢复？

```python
# 初始化时注入 checkpointer
def init_compiled_agent_graph(checkpointer: BaseCheckpointSaver) -> None:
    global _compiled, _checkpointer
    _checkpointer = checkpointer
```

## 3. 数据库设计

### Q: 为什么 TaskEvent 用 seq 字段？

**原因**：SSE 增量推送需要有序序号

```sql
CREATE TABLE task_events (
    id INTEGER PRIMARY KEY,
    task_id TEXT,
    seq INTEGER,  -- 有序序号，按任务独立
    module TEXT,
    kind TEXT,
    payload_json TEXT
);

-- SSE 查询：增量获取
SELECT * FROM task_events
WHERE task_id = ? AND seq > ?
ORDER BY seq
LIMIT 200
```

### Q: 为什么不直接用自增 ID？

| ID | seq |
|----|-----|
| 全局自增 | 任务内自增 |
| 全局唯一 | 按任务独立 |
| 无法支持增量查询 | 支持 `seq > last_seq` |

### Q: 会话黑板如何持久化？

```python
# backend/app/shared/blackboard.py

async def flush_blackboard_from_graph_checkpoint(
    graph, config: dict, *, session_id: str
) -> None:
    """任务结束后把 checkpoint 终态黑板刷回会话，供下一任务继承"""

    vals = dict(snap.values) if snap and snap.values else {}

    cap = get_settings().session_blackboard_max_notes

        await write_session_blackboard(db, session_id, notes, max_notes=cap)
```

## 4. 前端架构

### Q: 为什么用 Zustand 而不是 Redux？

| 特性 | Zustand | Redux |
|------|---------|-------|
| 样板代码 | 少 | 多 |
| 学习曲线 | 低 | 高 |
| 性能优化 | 自动 | 需 memo |
| 适用场景 | 中小规模 | 复杂状态 |

ForgeAgent 场景：前端状态不复杂，Zustand 足够。

### Q: SSE 在前端如何处理？

```typescript
// useTaskEvents.ts
const useTaskEvents = (taskId: string) => {
  useEffect(() => {
    const es = new EventSource(`/api/v1/tasks/${taskId}/events/stream`);

    es.addEventListener('plan_created', (e) => {
      store.updatePlan(JSON.parse(e.data));
    });

    es.addEventListener('tool_call', (e) => {
      store.appendToolCall(JSON.parse(e.data));
    });

    es.addEventListener('llm_stream_delta', (e) => {
      store.appendLlmDelta(JSON.parse(e.data));
    });

    return () => es.close();
  }, [taskId]);
};
```

### Q: 熔断器模式实现？

```python
# backend/app/core/circuit_breaker.py

class CircuitState(str, Enum):
    CLOSED = "closed"      # 正常
    OPEN = "open"          # 熔断
    HALF_OPEN = "half_open" # 半开探测

class CircuitBreaker:
    def before_call(self):
        """若当前不可执行则抛出 CircuitOpenError"""
        if self._state == CircuitState.OPEN:
            elapsed = (now - self._opened_at).total_seconds()
            if elapsed >= self.recovery_timeout_sec:
                self._state = CircuitState.HALF_OPEN  # 超时后进入半开
                return
            raise CircuitOpenError(f"熔断器已打开，{int(self.recovery_timeout_sec - elapsed)}s 后可重试")

    def record_success(self):
        """成功: 关闭熔断并清零失败计数"""
        self._state = CircuitState.CLOSED
        self._failure_count = 0

    def record_failure(self):
        """失败: 累加计数，达到阈值则打开"""
        if self._state == CircuitState.HALF_OPEN:
            self._state = CircuitState.OPEN  # 探测失败，再次打开
            return
        self._failure_count += 1
        if self._failure_count >= self.failure_threshold:
            self._state = CircuitState.OPEN
```

**状态转换图**：
```
CLOSED ──[失败×10]──▶ OPEN
  ▲                      │
  │                      │ 超时 60s
  │                      ▼
  │                   HALF_OPEN
  │                      │
  └──[探测成功]────────────┘
         ▲
         │ [探测失败]
         └──────────▶ OPEN
```
