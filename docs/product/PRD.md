# ForgeAgent 产品需求文档（PRD）

## 1. 产品概述

**ForgeAgent** 是一款面向开发与使用场景的 **AI Agent 应用**（非通用编排框架）：在单一产品内提供 **规划、记忆、工具、执行** 四类能力模块，采用 **Plan-and-Execute**（先规划后执行）的认知循环；支持 **模块化** 与可选扩展（**MCP**、**Skills 约定**、后续里程碑的 **多 Agent**）；并提供 **简洁现代的前端**，用于监控单次与多次任务执行。产品解决的核心问题是：在不过度定制底层协议的前提下，让团队 **敏捷迭代** Agent 能力，使功能反应 **快速、路径清晰、可验收**。

**MVP 一句话边界**（与调研结论一致）：**单 Agent + 显式规划循环 + 会话级记忆 + 工具/MCP 注册表 + 可观测执行**；多 Agent、重租户 SaaS、完整长期记忆治理等 **不纳入 MVP**。

## 2. 目标用户

| 用户类型 | 特征 | 主要诉求 |
|----------|------|----------|
| **Agent 应用开发者** | 需要在本机或小团队环境内调试、扩展 Agent；关注可测试、可观测、类型与边界清晰 | 少造轮子、快试想法、能对照业界抽象（规划/记忆/工具/执行）裁剪实现 |
| **Agent 使用者（轻量）** | 通过前端发起任务、查看步骤与结果，不直接改代码 | 看得懂「先规划再执行」、能确认失败点、界面不臃肿 |

## 3. 功能列表

### 必须做（MVP）

| 功能 | 描述 | 验收标准 |
|------|------|----------|
| **四大能力模块（应用内可见）** | 在代码与产品叙事上区分：**规划**（生成/更新任务计划）、**记忆**（当前会话上下文与短期状态）、**工具**（注册与调用）、**执行**（按计划逐步执行并写回状态）。 | 文档或内置说明中能对应到四个模块；单次任务流中可指出当前处于哪一模块行为（至少通过日志/前端步骤展示）。 |
| **Plan-and-Execute 主循环** | 执行任务前生成可结构化展示的计划（步骤列表或等价结构）；执行阶段按步骤推进，允许在失败或信息不足时 **重规划**（次数或条件可配置上限）。 | 前端或日志中能看到「计划 → 执行 →（可选）再计划」；同一任务 ID 下计划版本可区分或可追溯。 |
| **单 Agent 闭环** | MVP 仅保证 **一个主 Agent** 完成从规划到执行的全流程；不实现多 Agent 路由与冲突解决。 | 端到端用例（如：调用至少一个工具完成子目标）在单 Agent 下可跑通；无多角色并行协作需求。 |
| **工具与 MCP 注册** | 提供 **统一工具注册表**：本地工具与通过 **MCP** 暴露的工具均经同一入口声明名称、描述与权限范围（只读/可写等最小分级即可）。 | 至少接入 **一种** MCP Server 或等价模拟；调用前能在列表中看到工具来源（内置 / MCP）。 |
| **Skills 的 MVP 定位** | Skills 不作为第二套运行时：**作为约定目录或预置资源包**（例如固定文件夹结构 + 元数据），加载后映射到已有工具/提示模板，与 MCP 共用「工具注册」心智。 | 存在一个示例 Skill：加载后新增或覆盖一组可调用的工具/模板，且与 MCP 工具在同一注册表中展示。 |
| **会话级记忆** | 默认记忆为 **当前会话**（任务线程）内的消息与中间状态；**不**在 MVP 承诺跨用户多租户隔离与合规级「遗忘」流程。 | 同一会话内多轮对话能引用前文；刷新或新会话不强制保留（行为明确文档化）；可选：预留向量存储接口但不强制实现完整 RAG。 |
| **可观测执行（前端 + 日志）** | **前端**：任务列表、当前任务详情、步骤/事件时间线、错误与高亮状态。**后端**：结构化事件流或等价日志，便于对接后续 OpenTelemetry（MVP 可先文件或内存缓冲）。 | 用户在前端能看完一次完整执行过程；开发者能从日志或导出文件中 **按任务 ID 回放** 主要步骤；失败时能看到失败步骤与错误信息。 |
| **简洁现代 UI** | 布局清晰、信息层级分明；监控页以时间线或步骤列表为主，避免复杂低代码编排画布。 | 在 1920×1080 与常见笔记本宽度下无严重错位；核心监控路径（进入任务 → 看步骤 → 看结果）≤3 次主要点击。 |

## 4. 页面结构

- **首页 / 仪表盘**  
  - 最近任务、快捷发起任务入口（若 MVP 需要）。

- **任务列表**  
  - 全部任务：状态（进行中 / 成功 / 失败）、时间、摘要。

- **任务详情（监控核心）**  
  - 计划展示（步骤列表）  
  - 执行时间线（每步：模块标签、输入输出摘要、工具调用名、耗时）  
  - 错误与重试/重规划记录  
  - 原始日志折叠区（可选）

- **设置（最小）**  
  - MCP 连接配置（URL/命令/环境变量占位）  
  - Skills 路径或启用开关  
  - 与模型/API Key 相关的 **仅客户端或后端安全存储**说明（见非功能需求）

- **关于 / 帮助**  
  - MVP 边界说明（单 Agent、会话记忆、何为 Skills）

## 5. 界面设计要求

- **整体风格**：简洁现代；以 **数据与步骤可读性** 优先，减少装饰性组件；中文界面与等宽字体用于日志/JSON 片段。
- **色彩要求**：中性背景 + 单一强调色用于进行中/成功/失败状态；避免高饱和多色块干扰排障。
- **交互要求**：步骤时间线支持展开折叠；长内容（工具返回）默认摘要 + 展开全文；关键操作（如重新执行）需二次确认或明确后果说明。

## 6. 数据存储

- **需要存储什么数据**  
  - 任务元数据：ID、创建时间、状态、摘要。  
  - 会话内消息与执行事件（步骤、工具调用名、错误栈摘要）。  
  - MCP/Skills 配置项（非密钥明文时可存本地配置表）。  
  - 用户若登录：账号标识（MVP 可单机单用户则弱化）。

- **本地存储还是后端数据库**  
  - **MVP 推荐**：单机或小团队场景下 **本地持久化**（如嵌入式数据库或本地文件）即可；若已存在后端服务，则使用 **后端数据库** 统一存任务与事件，前端通过 API 拉取。二者选其一并在实现中写死，避免双写。

---

## 7. 非功能性需求

- **性能**：任务列表首屏加载在本地网络下 **≤ 2s**（数据量按百级任务计）；单任务详情页渲染 **≤ 1s**（不含超大日志全量渲染时可分页或虚拟列表）。
- **兼容性**：前端支持 **最新两个大版本** 的 Chrome / Edge / Firefox；不要求 IE。
- **安全**：LLM 与 MCP 使用的 **API Key、密钥不得写入前端打包产物或公开仓库**；通过环境变量或服务端注入；日志中对密钥类字段脱敏。
- **可维护性**：模块边界与「不做什么」在 README 或帮助页可见，降低与 LangGraph 等全量框架的功能预期重叠（见调研文档结论）。

---

## 8. 明确不做什么（降低预期）

- 不做 **通用多租户 SaaS** 与完整计费、权限矩阵（MVP）。  
- 不做 **可视化拖拽式** 复杂工作流编排器（除非单独立项）。  
- 不在 MVP 内承诺 **多 Agent 生产级** 协作与冲突解决。  
- 不把 **完整认知架构**（信念、心理理论等）作为默认内置——仅 **Plan-and-Execute + 显式状态** 为默认路径，其余以插件或后续里程碑形式出现。
